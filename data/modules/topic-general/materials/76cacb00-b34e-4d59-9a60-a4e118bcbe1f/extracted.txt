Anatomy and terminology of linked lists
• A LinkedList consists of a connected sequence of Link/Node objects
• Each link/node contains a reference to a datum and a reference to the next node in the 
sequence. 
• The datum is just an object. It could be just an int or a string, or it could contain a key/data pair.
• The last node in the sequence refers to None as its next node
• The sequence of nodes is accessed via a “header” object that contains a reference to the first 
node in the sequence.

More terminology
• A node’s successor is the next node in the sequence
• The last Link has no successor
• A node’s predecessor is the previous node in the sequence
• The first Link has no predecessor
• A LinkedList’s length is the number of nodes in it
• A linked list may be empty (contain no nodes)

Deleting a Link from a LinkedList
• In order to delete a Link from a LinkedList, you have to change the pointer in 
its predecessor
• This is slightly tricky, because you can’t follow a pointer backwards
• Deleting the first Link in a list is a special case, because the Link’s predecessor 
is the list header

Real world Application – Task Scheduling
Task Scheduling : 
Linked lists are often used in real-world task scheduling, especially in operating systems and 
other environments where efficient management of tasks or processes is critical.
Advantages of Linked Lists in Task Scheduling
Efficient Insertion/Deletion: Linked lists allow for quick insertion and deletion of tasks, which is 
essential in environments where tasks frequently enter and exit the queue.
Dynamic Sizing: No pre-allocation of memory is needed, making linked lists ideal for systems 
with fluctuating task loads.

Linked-list implementation details
• With a linked-list representation, overflow will not happen (unless you exhaust 
memory, which is another kind of problem)
•  Underflow can happen, and should be handled the same way as for an array 
implementation
• When a Link’s datum is popped from a list implementation of a stack, and the 
Link references an object, the reference (the pointer in the Link) does not 
need to be set to None
• Unlike an array implementation, it really is removed—the garbage collector can no longer 
get to it from the linked list
• Hence, garbage collection will occur as appropriate

Can we implement Queues with LinkedLists?
• In a queue, insertions occur at one end, deletions at the other end
• Operations at the front of a LinkedList are O(1), but at the other end they are 
O(n)
• Why?
• BUT: there is a simple way to use a LinkedList list to implement both 
insertions at the front and deletions at the end in O(1) time
• You always need a pointer to the first thing in the list
• You can also keep an additional pointer to the last Link in the list
• This is called a double-ended list

Queue implementation comparison
• With an array implementation:
• you can have both overflow and underflow
• you should set deleted elements to None
• With a linked-list implementation:
• You can have underflow
• Overflow is a global out-of-memory condition
• There is no reason to set deleted elements to None
